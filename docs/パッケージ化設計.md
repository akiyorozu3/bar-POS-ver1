# バーPOS パッケージ化 設計ドキュメント

> ステータス: **v0.1（ドラフト・随時更新）**
> 最終更新: 2026-07-02
> これは「複数のバーへ販売できる商品」にしていくための設計の合意メモです。機能拡張に合わせて更新します。

---

## 1. ゴール

いまの「1店専用」のバーPOSを、**複数のバーに導入できるパッケージ商品**にする。
機能拡張を続けながら、以下を満たす土台を作る。

- 開発・検証で**本番データを汚さない**
- 2店目以降を**複製で素早く立ち上げられる**
- 全店へ**安全に更新を配信できる**

---

## 2. 確定した方針

| 項目 | 決定 | 理由 |
|---|---|---|
| テナント方式 | **A方式：店ごとに別Firebaseプロジェクト** | データ完全隔離・ルール単純・既存コードを活かせる |
| データ所有/課金 | **各店が所有**（各店の無料枠内） | 開発者にインフラ費・障害責任が乗らない。「データは店のもの」は営業上も強い |
| 開発者の権限 | 各店プロジェクトに**編集者/オーナーで参加**して更新を配信 | 更新責任を持ちつつ、所有は店側 |
| 販売形態 | **売り切り／月額 どちらも可（未定）** | A方式はどちらとも相性可 |
| オンボーディング | **当面は手動＋開店スクリプト**（セルフサービスは将来） | 少数店ならこれで十分 |

### 不採用：B方式（マルチテナント同居）
全docに `storeId` を付け1プロジェクトに同居する案。管理は1箇所で済むが、**全docの改修・ルール全書き換え・取り違え時の情報漏洩リスク**を常に背負うため、本用途では不採用。

## 3. 環境構成

```
[dev]      Firebaseエミュレータ（ローカル・無料）   … 日々の開発・自動検証。本番に一切触れない
[staging]  開発者所有の1プロジェクト               … 新機能のQA・営業デモ
[prod]     店ごとに1プロジェクト（テンプレを複製）    … 各バーの本番（例: bar-pos-b1993）
```

- 全環境で**同一コードベース**。違いは「ビルド時に読む鍵・config」だけ。
- 切り替えは Vite のモード（`.env.emulator` / `.env.staging` / `.env.production`）で行い、
  **どのDBに繋ぐかはビルド時に確定**させる（「テストのつもりが本番」を構造的に防ぐ）。

---

## 4. リポジトリ側でやる土台整備

### Phase 0：エミュレータ導入 ← 着手中
- `firebase.json` に emulators / firestore 設定を追加
- `src/lib/firebase.ts` が `VITE_USE_EMULATOR=true` のときエミュレータに接続
- `npm run emu`（エミュレータ起動）／`npm run dev:emu`（アプリをエミュレータ接続で起動）
- `scripts/seed-emulator.mjs`（オーナー/スタッフ発行＋サンプルデータ投入）
- **効果：Claude/開発の検証が本番Firestoreを触らなくなる**

### Phase 1：多店対応の土台（機能拡張の前にやる）
1. **config外出し** … 店名・税モード・手数料/バック初期値・カテゴリ・卓定型を `settings/config` ＋ env に。今ベタ書きの `owner@bar-pos.local` もここへ
2. **ロールを custom claims 化** … 「メール文字列一致」をやめ Firebase のカスタムクレームで owner/staff 判定。ルールも `request.auth.token.role == 'owner'` に
3. **パスワード見直し（販売前必須）** … `password`/`000000` を廃止。店ごと初期PW＋初回変更必須

### Phase 2：複製で開店できる状態
- **開店スクリプト**（1コマンドで：ルール/インデックス投入 → メニュー等シード → オーナー発行 → 初回デプロイ）
- **staging プロジェクト**を用意し、リリース前QAの置き場に

### Phase 3：機能拡張
- 以降の新機能はすべてこの土台の上で。dev(エミュレータ)→staging→各店prod の順で流す

## 5. 運用（A方式の継続コストと対策）

### 更新の全店配信
店＝別プロジェクトなので、同じビルドを各店Hostingへ配る。

- **店台帳** `stores.json`（例）を1つ持つ：
  ```json
  [
    { "name": "◯◯バー", "projectId": "bar-pos-b1993", "since": "2026-07", "plan": "..." }
  ]
  ```
  ※ 各店の鍵/秘匿情報は**リポジトリに入れない**（別管理）。台帳はメタ情報のみ。
- `npm run deploy:all` で台帳をループして各店へ配信（Phase 2 で用意）

### スキーマ変更（マイグレーション）
docの形を変える改修は全店へ流す必要がある。
→ **マイグレーションを1本ずつスクリプト化**し、台帳ループで流す運用にする。
   今のうちからこの型にしておくと後が楽。

## 6. 販売前チェックリスト（商品化の必須項目）

- [ ] 弱いパスワードの廃止（店ごと初期PW＋変更必須）
- [ ] ロールを custom claims 化（メール一致判定の撤廃）
- [ ] Firestore ルールの再監査（別プロジェクト前提でシンプルに）
- [ ] `.env`・秘匿情報がリポジトリに含まれないことの確認
- [ ] 店ごとの config 外出し完了
- [ ] 開店スクリプト・全店デプロイ・マイグレーションの手順整備
- [ ] バックアップ/データエクスポート手段（店へ渡せる形）

---

## 7. 現状の技術メモ（引き継ぎ用）

- スタック: React 18 + TS + Vite 5 + Zustand 4 + Firebase v10（Firestore + Auth email/password）/ Firebase Hosting
- ロール判定（現状・要変更）: `owner@bar-pos.local` を owner、他を staff（`src/lib/authConfig.ts` / `firestore.rules`）
- コレクション: transactions / menus / casts / settings / tables / closures / punches / payouts
- Hosting: `index.html` は no-cache 済み（旧ビルドのキャッシュで不整合が起きた対策）
